**1. `oc-modal`：基础组件二次封装**
- 代码位置：[oc-modal/type.ts](file:///Users/bytedance/Projects/ocean-ui-need/packages/atom/packages/oc-components/oc-modal/type.ts)、[oc-modal/index.vue](file:///Users/bytedance/Projects/ocean-ui-need/packages/atom/packages/oc-components/oc-modal/index.vue)。
- 这个组件不是从零实现弹窗，而是基于底层 `@tools-ui/vue3` 的 `Modal` 再封装，代码里通过 `import { Modal as OModal } from "@tools-ui/vue3"` 引入底层能力。
- 类型层面通过 `TModalProps = InstanceType<typeof OModal>["$props"]` 复用底层弹窗类型，避免重复维护底层 `teleportTo` 等参数。
- 对外 API 重新定义为 `IOcModalProps`，包括 `show`、`width`、`title`、`showClose`、`maskClosable`、`showFooter`、`confirmLoading` 等，更贴合 Ocean UI 的使用习惯。
- 事件层面通过 `IOcModalEmits` 明确定义 `update:show`、`cancel-click`、`confirm-click`、`visible-change` 等事件，把弹窗状态变化和用户操作暴露给业务方。
- 模板里底层使用 `<o-modal>`，但 footer 区域默认渲染 Ocean 自己的 `oc-button`，并支持 `footer` 插槽覆盖，这就是“默认体验统一 + 允许业务自定义”的典型封装。

可以重点看这里：[oc-modal/index.vue:L2-L59](file:///Users/bytedance/Projects/ocean-ui-need/packages/atom/packages/oc-components/oc-modal/index.vue#L2-L59)。

这个组件的封装思路是：

- `底层能力复用`：不重复造弹窗，而是复用 `OModal`。
- `统一设计规范`：默认按钮、圆角、间距、footer 样式由 Ocean UI 控制。
- `状态受控`：通过 `show` 和 `update:show` 支持 `v-model:show`。
- `扩展点开放`：通过 `header`、`footer`、`default` 插槽支持业务自定义。
- `兼容旧 API`：同时触发 `update:show` 和 `update-show`，照顾历史使用方式。
- `埋点内聚`：通过 `usePerform` 统一处理 `visible-change`、按钮点击等埋点。

面试可以这样讲：

> `oc-modal` 是一个典型的基础组件二次封装。它底层复用 `@tools-ui/vue3` 的 `Modal`，但在 Ocean UI 层重新定义了更符合业务使用习惯的 Props、事件和默认样式。比如显隐通过 `show` 控制，确认和取消通过 `confirm-click`、`cancel-click` 通知外部，footer 默认使用统一的 `oc-button`，同时又通过插槽允许业务自定义。这样既保证一致性，又保留扩展能力。

**2. `create-filter-bar`：配置驱动业务组件**
- 代码位置：[create-filter-bar/type.ts](file:///Users/bytedance/Projects/ocean-ui-need/packages/atom/packages/ad-components/create-filter-bar/type.ts)、[create-filter-bar/index.vue](file:///Users/bytedance/Projects/ocean-ui-need/packages/atom/packages/ad-components/create-filter-bar/index.vue)。
- 这个组件的核心不是固定写死几个筛选项，而是通过 `config` 配置数组驱动筛选栏渲染。
- 类型里 `IFilterItemConfig` 定义了每个筛选项的能力，包括 `key`、`type`、`label`、`placeholder`、`options`、`multiple`、`filterable`、`renderFn` 等。
- `type` 支持 `"select" | "search" | "cascader" | "render"`，说明这个组件把常见筛选场景抽象成几类渲染模式。
- 模板里通过 `v-for="item of config"` 遍历配置，再根据 `item.type` 渲染 `oc-select`、`oc-input`、`oc-cascader` 或自定义 `render-component`。
- 数据层面通过 `value: TModel` 和 `update:value` 实现外部受控，组件内部不直接拥有业务查询状态，而是负责收集变化并通知外部。
- 交互层面通过 `handleChange` 统一处理不同筛选项变化，同时 `debounce` 100ms 避免频繁触发。

**封装方法总结**
- `先定义类型`：每个组件都有独立 `type.ts`，用接口和枚举明确组件 API，比如 `IOcModalProps`、`IFilterItemConfig`、`TOcMediaUploadItem`。
- `再定义默认值`：在 `index.vue` 中通过 `withDefaults(defineProps())` 给出默认行为，保证组件开箱即用。
- `用事件隔离业务`：组件内部处理 UI 和交互，业务动作通过 `emit` 抛出，比如上传组件不直接调接口，而是 emit `upload`。
- `用插槽开放展示`：默认 UI 保持一致，但 `header`、`footer`、`append`、`tip`、`desc` 等插槽允许业务定制。
- `用 computed 建模状态`：例如 `showSelectCard`、`selectButtonDisabled`、`status`、`extraClassComp`，把复杂状态集中计算。
- `用子组件拆复杂 UI`：上传组件把选择卡片、预览卡片、文件卡片拆开，避免单文件过大。
- `统一埋点和样式前缀`：多个组件都接入 `usePerform` 和 `useCssPrefix`，保证组件库级一致性。

- 技术重点：数据格式转换、图表配置抽象、tooltip、legend、坐标轴、空态、异常数据兜底。
- 业务价值：报表和投放分析场景需要大量图表，统一图表组件可以保证数据可视化风格一致。
- 面试可讲：图表组件的重点是把复杂配置封装起来，让业务方只关注数据和少量配置。

可以这样回答：

> 图表类组件主要解决的是数据可视化复用问题。业务方通常不希望直接写复杂图表配置，所以组件库会把坐标轴、tooltip、legend、空态等通用逻辑封装起来，对外暴露更简单的数据和配置接口。

**九、和 AI 单测 Skill 的结合**
- 输入类组件：AI 可以根据 `Props`、`v-model`、`Emits` 自动生成输入、清空、禁用、边界值测试。
- 弹层类组件：AI 可以生成打开、关闭、确认、取消、遮罩点击、loading 状态测试。
- 展示类组件：AI 可以生成有数据、空数据、loading、字段缺失、插槽自定义测试。
- 上传类组件：AI 可以生成文件格式错误、上传成功、上传失败、删除、预览等测试用例。
- 导航类组件：AI 可以生成菜单选中、展开收起、点击事件、不同菜单层级渲染测试。
- 业务组件：AI 可以基于 Demo、mock 数据和 `type.ts` 生成更贴近业务流程的测试。

可以这样回答：

> 我们做 AI 单测 Skill 时，并不是让 AI 泛泛地写测试，而是结合组件类型设计测试策略。比如输入类组件重点测 `v-model` 和事件，弹窗类组件重点测打开关闭和确认取消，上传类组件重点测异步状态和错误处理，展示类组件重点测空态、loading 和插槽。这样生成的测试才更贴合组件行为。

**十、和 AI Code Review Skill 的结合**
- 输入类组件：检查类型和值是否一致，事件触发是否合理，是否遗漏禁用/清空状态。
- 弹层类组件：检查关闭逻辑是否完整，异步 loading 是否会卡死，事件命名是否清晰。
- 展示类组件：检查空数据、字段缺失、长文本、插槽兜底是否处理。
- 上传类组件：检查文件校验、失败兜底、状态恢复、资源释放是否完整。
- 业务组件：检查业务逻辑是否写死、是否缺少扩展点、是否破坏已有 API。
- 公共组件：检查样式是否污染、类型是否过宽、是否引入不必要副作用。

可以这样回答：

> AI Code Review Skill 的价值在于把组件库里常见的 Review 经验规则化。比如看到上传组件，会重点检查异步状态和错误兜底；看到弹窗组件，会检查关闭和确认事件；看到展示组件，会检查空态和异常数据；看到公共组件，会关注 API 兼容性和样式隔离。
